<h1 id="errores-y-validaciones"><b>Errores y validaciones</b></h1>
<p class="no-margin"></p>
<p class="no-margin">Las respuestas de error de la API pública siguen siempre la misma forma
base, independientemente del recurso. Esta guía documenta los códigos HTTP
que verás, el formato JSON, y cómo distinguir entre los distintos tipos.</p>
<p class="no-margin"></p>
<h2 id="forma-general-de-la-respuesta"><b>Forma general de la respuesta</b></h2>
<p class="no-margin"></p>
<p class="no-margin">Toda respuesta de error tiene como mínimo:</p>
<p class="no-margin"></p>
<pre><code class="language-json">{
  &quot;statusCode&quot;: 400,
  &quot;message&quot;: &quot;Mensaje legible que explica el problema&quot;
}</code></pre>
<p class="no-margin"></p>
<p class="no-margin">Puede incluir además:</p>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin"><b><code>type</code></b> — categoría del error cuando aplica.</p></li>
<li><p class="no-margin"><b><code>errors</code></b> — array con detalles por campo en errores de validación
(solo en <code>400 ValidationError</code>).</p></li>
</ul>
<p class="no-margin"></p>
<p class="no-margin"><code>Content-Type</code> siempre es <code>application/json</code>.</p>
<p class="no-margin"></p>
<h2 id="cdigos-http"><b>Códigos HTTP</b></h2>
<p class="no-margin"></p>
<div class="intercom-interblocks-table-container">
<table>
<thead>
<tr>
<th>Código</th>
<th>Significado típico</th>
</tr>
</thead>
<tbody>
<tr>
<td><code>400 Bad Request</code></td>
<td>El cuerpo o los parámetros no cumplen el contrato. Si es un fallo de validación de campos, viene con <code>errors[]</code> (ver más abajo).</td>
</tr>
<tr>
<td><code>401 Unauthorized</code></td>
<td>Falta credencial, está expirada o es inválida. Ver <a href="https://help.facturadirecta.com/es/articles/13538825">Autenticación</a>.</td>
</tr>
<tr>
<td><code>403 Forbidden</code></td>
<td>Credencial válida pero sin los scopes necesarios, o sin acceso a la empresa indicada en el path.</td>
</tr>
<tr>
<td><code>404 Not Found</code></td>
<td>El recurso no existe o no pertenece a la empresa del path.</td>
</tr>
<tr>
<td><code>409 Conflict</code></td>
<td>La operación choca con el estado actual: identificador duplicado, borrado de un recurso con dependencias, etc.</td>
</tr>
<tr>
<td><code>500 Internal Server Error</code></td>
<td>Error inesperado del servidor. No es un error del cliente; conviene reintentar tras un retraso.</td>
</tr>
</tbody>
</table>
</div>
<p class="no-margin"></p>
<h2 id="errores-de-validacin-400"><b>Errores de validación (<code>400</code>)</b></h2>
<p class="no-margin"></p>
<p class="no-margin">Cuando el body o los parámetros fallan el JSON Schema, la respuesta incluye
un array <code>errors</code> con un objeto por cada problema:</p>
<p class="no-margin"></p>
<pre><code class="language-json">{
  &quot;statusCode&quot;: 400,
  &quot;message&quot;: &quot;Validation failed&quot;,
  &quot;type&quot;: &quot;ValidationError&quot;,
  &quot;errors&quot;: [
    {
      &quot;path&quot;: &quot;content.main.fiscalId&quot;,
      &quot;message&quot;: &quot;Formato de NIF/CIF inválido para España&quot;
    },
    {
      &quot;path&quot;: &quot;content.main.lines.0.unitPrice&quot;,
      &quot;message&quot;: &quot;Debe ser un número mayor que 0&quot;
    }
  ]
}</code></pre>
<p class="no-margin"></p>
<p class="no-margin">Cada entrada del array localiza el campo problemático y explica qué falla.
Para integraciones que muestran el error al usuario final, basta con
mostrar el <code>message</code> de cada entrada; el <code>path</code> ayuda al developer a
depurar.</p>
<p class="no-margin"></p>
<h2 id="errores-de-negocio-400-409"><b>Errores de negocio (<code>400</code>, <code>409</code>)</b></h2>
<p class="no-margin"></p>
<p class="no-margin">Algunos errores no son fallos de schema sino reglas de negocio:</p>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin">Crear una factura con una serie ya consumida.</p></li>
<li><p class="no-margin">Borrar un contacto que tiene documentos asociados.</p></li>
<li><p class="no-margin">Modificar un presupuesto convertido en factura.</p></li>
</ul>
<p class="no-margin"></p>
<p class="no-margin">Estos errores vienen como <code>400</code> o <code>409</code> (según el caso) con un <code>message</code>
descriptivo. Cuando el mensaje incluye datos relevantes (ID de documento
bloqueante, valor duplicado), se devuelven en campos adicionales del JSON.</p>
<p class="no-margin"></p>
<h2 id="errores-de-autenticacin-y-autorizacin-401-403"><b>Errores de autenticación y autorización (<code>401</code>, <code>403</code>)</b></h2>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin"><b><code>401</code></b> — el token o la API key son rechazados antes de evaluar
permisos. Casos típicos: token expirado, API key revocada, header mal
formado.</p></li>
<li><p class="no-margin"><b><code>403</code></b> — la credencial es válida pero le falta algo:</p>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin">Scopes insuficientes para la operación (por ejemplo, intentar <code>POST</code> con scope <code>read</code>).</p></li>
<li><p class="no-margin">La empresa del path no es accesible para esta credencial.</p></li>
</ul>
<p class="no-margin"></p></li>
</ul>
<p class="no-margin"></p>
<p class="no-margin">Para detalles del flujo de auth y rotación de tokens, ver
<a href="https://help.facturadirecta.com/es/articles/13538825">Autenticación</a>.</p>
<p class="no-margin"></p>
<h2 id="recursos-no-encontrados-404"><b>Recursos no encontrados (<code>404</code>)</b></h2>
<p class="no-margin"></p>
<p class="no-margin"><code>404</code> significa siempre <b>una de estas dos cosas</b>:</p>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin">El ID no existe en la empresa indicada.</p></li>
<li><p class="no-margin">El ID existe pero pertenece a otra empresa (la API no revela que el
recurso existe en otro tenant).</p></li>
</ul>
<p class="no-margin"></p>
<p class="no-margin">Trata ambos casos como &quot;no existe para mí&quot; en tu integración.</p>
<p class="no-margin"></p>
<h2 id="estrategia-recomendada-de-manejo"><b>Estrategia recomendada de manejo</b></h2>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin"><b>Diferencia errores del cliente (4xx) de errores del servidor (5xx).</b>
Los 4xx no deben reintentarse sin cambiar el input; los 5xx sí, con
backoff exponencial.</p></li>
<li><p class="no-margin"><b>Para <code>400 ValidationError</code>, recorre <code>errors[]</code></b> y muestra al
usuario los <code>message</code> específicos en vez del <code>message</code> general.</p></li>
<li><p class="no-margin"><b>Idempotencia.</b> Repetir un <code>POST /contacts</code> con el mismo
<code>fiscalId</code> da <code>409 Conflict</code>, no duplicado. Aprovecha esto para
reintentos seguros en sincronizaciones.</p></li>
<li><p class="no-margin"><b>No deduzcas estado a partir del código HTTP solo.</b> Combina código y
contenido (<code>type</code>, <code>message</code>) para clasificar el error en tu lógica.</p></li>
</ul>
<p class="no-margin"></p>
<h2 id="ejemplo-completo"><b>Ejemplo completo</b></h2>
<p class="no-margin"></p>
<p class="no-margin">Petición que viola dos validaciones (CIF mal formado y línea con precio
negativo):</p>
<p class="no-margin"></p>
<pre><code class="language-http">HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  &quot;statusCode&quot;: 400,
  &quot;message&quot;: &quot;Validation failed&quot;,
  &quot;type&quot;: &quot;ValidationError&quot;,
  &quot;errors&quot;: [
    {
      &quot;path&quot;: &quot;content.main.fiscalId&quot;,
      &quot;message&quot;: &quot;Formato de NIF/CIF inválido para España&quot;
    },
    {
      &quot;path&quot;: &quot;content.main.lines.0.unitPrice&quot;,
      &quot;message&quot;: &quot;Debe ser un número mayor que 0&quot;
    }
  ]
}</code></pre>
<p class="no-margin"></p>
<p class="no-margin">Y conflicto al borrar un contacto con factura asociada:</p>
<p class="no-margin"></p>
<pre><code class="language-http">HTTP/1.1 409 Conflict
Content-Type: application/json

{
  &quot;statusCode&quot;: 409,
  &quot;message&quot;: &quot;No se puede borrar el contacto: tiene 3 facturas asociadas&quot;
}</code></pre>
