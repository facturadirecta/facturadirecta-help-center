<h1 id="autenticacin"><b>Autenticación</b></h1>
<p class="no-margin"></p>
<p class="no-margin">La API pública de FacturaDirecta soporta dos métodos de autenticación,
<a href="#oauth2">OAuth2</a> y <a href="#api-key">API key</a>. Cada operación acepta ambos; tú
eliges el que mejor encaje con tu integración. Ambos comparten los mismos
scopes y operan sobre el mismo conjunto de endpoints.</p>
<p class="no-margin"></p>
<h2 id="endpoint-base"><b>Endpoint base</b></h2>
<p class="no-margin"></p>
<p class="no-margin">Todas las URLs de la API pública en producción tienen el prefijo:</p>
<p class="no-margin"></p>
<pre><code>https://app.facturadirecta.com/api</code></pre>
<p class="no-margin"></p>
<p class="no-margin">A partir de ahí, los paths que se documentan en cada recurso (por ejemplo
<code>/{companyId}/contacts</code>) se concatenan a esta base.</p>
<p class="no-margin"></p>
<p class="no-margin">Recursos auxiliares:</p>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin"><b>Referencia OpenAPI navegable (Swagger UI):</b> <code>https://www.facturadirecta.com/api</code></p></li>
<li><p class="no-margin"><b>Especificación OpenAPI cruda (<code>openapi.json</code>):</b> <code>https://app.facturadirecta.com/openapi.json</code></p></li>
</ul>
<p class="no-margin"></p>
<h2 id="oauth2"><b>OAuth2</b></h2>
<p class="no-margin"></p>
<p class="no-margin">Recomendado cuando un usuario conecta su propia empresa de FacturaDirecta a
tu aplicación. Tú obtienes un token en su nombre y operas dentro de su
empresa.</p>
<p class="no-margin"></p>
<p class="no-margin">FacturaDirecta usa Keycloak como servicio de autorización, con realm
<code>facturadirecta</code>. El flujo es <code>authorizationCode</code> estándar:</p>
<p class="no-margin"></p>
<ol>
<li><p class="no-margin">Rediriges al usuario a la URL de autorización con <code>client_id=facturadirecta-api</code>
y los scopes que necesitas.</p></li>
<li><p class="no-margin">El usuario autentica y autoriza los scopes.</p></li>
<li><p class="no-margin">Keycloak redirige a tu callback con un <code>code</code>.</p></li>
<li><p class="no-margin">Intercambias <code>code</code> por <code>access_token</code> (y opcionalmente <code>refresh_token</code>)
en el endpoint de token.</p></li>
<li><p class="no-margin">Usas el <code>access_token</code> en el header <code>Authorization: Bearer &lt;token&gt;</code> para
cada llamada a la API.</p></li>
</ol>
<p class="no-margin"></p>
<p class="no-margin"><b>URLs del servicio de autorización en producción:</b></p>
<p class="no-margin"></p>
<div class="intercom-interblocks-table-container">
<table>
<thead>
<tr>
<th>Endpoint</th>
<th>URL</th>
</tr>
</thead>
<tbody>
<tr>
<td>Autorización</td>
<td><code>https://auth.facturadirecta.com/auth/realms/facturadirecta/protocol/openid-connect/auth</code></td>
</tr>
<tr>
<td>Token</td>
<td><code>https://auth.facturadirecta.com/auth/realms/facturadirecta/protocol/openid-connect/token</code></td>
</tr>

</tr>

</div>
<p class="no-margin"></p>
<p class="no-margin">La fuente autoritativa es el <code>openapi.json</code> público en
<code>https://app.facturadirecta.com/openapi.json</code>, sección
<code>components.securitySchemes.oAuth.flows.authorizationCode</code>.</p>
<p class="no-margin"></p>
<p class="no-margin"><b><code>client_id</code></b> es <code>facturadirecta-api</code> y <b>no requiere <code>client_secret</code></b>.</p>
<p class="no-margin"></p>
<p class="no-margin"><b><code>offline_access</code></b> como scope te permite obtener un <code>refresh_token</code> para
operar sin nueva intervención del usuario.</p>
<p class="no-margin"></p>
<h3 id="ejemplo-de-cabecera"><b>Ejemplo de cabecera</b></h3>
<p class="no-margin"></p>
<pre><code class="language-http">Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCIgOiAiSldUIi...</code></pre>
<p class="no-margin"></p>
<h2 id="api-key"><b>API key</b></h2>
<p class="no-margin"></p>
<p class="no-margin">Si tu integración solo necesita acceder a una empresa concreta que tú
administras, puedes generar una API key desde <b>Ajustes &gt; Integraciones</b>
en la propia empresa de FacturaDirecta, o crearla por API desde
<code>POST /apiKeys</code>.</p>
<p class="no-margin"></p>
<p class="no-margin">Al crear una API key le asignas los scopes que va a poder usar (los mismos
que están disponibles para OAuth2, salvo <code>offline_access</code> que para una API
key no tiene sentido).</p>
<p class="no-margin"></p>
<h3 id="ejemplo-de-cabecera"><b>Ejemplo de cabecera</b></h3>
<p class="no-margin"></p>
<pre><code class="language-http">facturadirecta-api-key: aB3cD9.xY7zN2qK4mP8rT5sJ1hF6gL0vW3uE</code></pre>
<p class="no-margin"></p>
<p class="no-margin">Formato: <code>&lt;prefijo de 6 caracteres&gt;.&lt;sufijo de 32 caracteres&gt;</code>. Es lo que
devuelve <code>POST /apiKeys</code> en el campo <code>apiKey</code> de la respuesta (solo
visible en la creación; tras eso, FacturaDirecta solo conserva el hash).
La API key se envía como header propio, <b>no</b> como <code>Authorization: Bearer</code>.</p>
<p class="no-margin"></p>
<h2 id="versin-de-la-api"><b>Versión de la API</b></h2>
<p class="no-margin"></p>
<p class="no-margin">Todas las peticiones aceptan el header <code>accept-version</code> con el número de
versión de la API. Si no lo envías, se usa la versión por defecto definida
por el servidor. Para producción se recomienda fijarlo explícitamente:</p>
<p class="no-margin"></p>
<pre><code class="language-http">accept-version: 1.0.9</code></pre>
<p class="no-margin"></p>
<h2 id="scopes"><b>Scopes</b></h2>
<p class="no-margin"></p>
<p class="no-margin">Cada operación de la API declara los scopes que requiere. Los scopes siguen
el patrón <code>&lt;recurso&gt;:&lt;acción&gt;</code>:</p>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin"><code>&lt;recurso&gt;:read</code> — permite leer (<code>GET</code>).</p></li>
<li><p class="no-margin"><code>&lt;recurso&gt;:write</code> — permite crear, modificar y borrar (<code>POST</code>, <code>PUT</code>, <code>DELETE</code>).</p></li>
</ul>
<p class="no-margin"></p>
<p class="no-margin">El conjunto exacto de scopes y su descripción está en el openapi público y
en cada página de recurso de esta documentación. Pide siempre los scopes
mínimos necesarios para la integración.</p>
<p class="no-margin"></p>
<h2 id="errores-de-autenticacin"><b>Errores de autenticación</b></h2>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin"><b><code>401 Unauthorized</code></b> — token expirado, ausente, mal formado o API key
inválida. Refresca el token (si tienes <code>refresh_token</code>) o regenera la
API key.</p></li>
<li><p class="no-margin"><b><code>403 Forbidden</code></b> — el token o la API key son válidos, pero les falta
algún scope para la operación. Revisa los scopes asignados.</p></li>
<li><p class="no-margin"><b><code>403 Forbidden</code> con <code>companyId</code> ajeno</b> — el token no tiene acceso a
la empresa indicada en el path. En OAuth, el usuario solo accede a
empresas en las que tiene rol; en API key, solo a la empresa en la que
se creó.</p></li>
</ul>
<p class="no-margin"></p>
<p class="no-margin">Para el formato general de errores ver <a href="https://help.facturadirecta.com/es/articles/15084452">Errores y validaciones</a>.</p>
<p class="no-margin"></p>
<h2 id="recomendaciones"><b>Recomendaciones</b></h2>
<p class="no-margin"></p>
<ul>
<li><p class="no-margin"><b>No registres tokens ni API keys en logs</b>, ni siquiera durante
desarrollo.</p></li>
<li><p class="no-margin"><b>Limita los scopes</b> al mínimo necesario. Si solo lees contactos, no
pidas <code>contacts:write</code>.</p></li>
<li><p class="no-margin"><b>Una API key por integración.</b> Si una integración deja de usarse,
rota la API key antes de archivarla.</p></li>
<li><p class="no-margin"><b>Para multi-tenant SaaS</b>, OAuth2 con <code>offline_access</code> es el patrón
correcto. API keys son por empresa y no escalan a &quot;una clave para
varias empresas&quot;.</p></li>
</ul>
